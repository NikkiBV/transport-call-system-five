# 🚚 Logistics Transport Forecasting System | Командный трек

## 📋 Описание проекта
Автоматизированная система прогнозирования отгрузок и генерации заявок на вызов транспорта. Решает задачу минимизации простоев складов и оптимизации логистических затрат через точное многошаговое прогнозирование.

## 📊 Метрики и Результаты (Лидерборд: 50%)
- **Трек:** `team`
- **Горизонт прогнозирования:** `10` шагов (по 30 мин)
- **Валидационная метрика (WAPE + Rbias):** `0.2674` (Цель: ≤ 0.290)
- **Архитектура модели:** Direct Multi-step Forecasting (отдельная модель на каждый шаг)
- **Обучено моделей:** 10 (LightGBM Regressor)

## 🏗️ Архитектура системы (Качество сервиса: 20%)
```
[Исторические данные .parquet] 
        ↓
[Feature Engineering: лаги, скользящие средние, временные флаги]
        ↓
[LightGBM Ensemble] → [Bias Correction & Clipping]
        ↓
[Decision Engine] → Заявки на транспорт / Алерты диспетчеру
```
- **Data Layer:** Парсинг, очистка, строгая хронологическая сортировка
- **ML Layer:** Прямое многошаговое прогнозирование без накопления ошибок рекурсии
- **Service Layer:** Конвертация прогноза в заявки: `trucks = ceil(forecast / capacity)`
- **Monitoring:** Валидация на временном сплите, расчёт `bias_correction_factor` для компенсации систематического смещения

## ⚙️ Бизнес-логика работы
- **Периодичность:** Прогноз генерируется каждые 30 минут (шаг данных)
- **Горизонт уверенности:** Шаги 1–4 (2ч вперёд) — для автоматического вызова. Шаги 5–10 — для предварительного планирования
- **Пороговые правила:** При `forecast > threshold` создаётся заявка. При высокой дисперсии отправляется алерт логисту
- **Обработка аномалий:** `np.clip(..., 0, None)` исключает отрицательные прогнозы. Калибровка смещения устраняет систематическую недо/переоценку объёмов

## 🤖 Модельный стек
- **Алгоритм:** LightGBM (`objective='regression'`, `metric='mae'`, `early_stopping=50`)
- **Признаки:** Временные (`hour`, `dayofweek`, `is_weekend`), лаги цели (1–8), скользящие средние (4, 8), категориальные ID маршрутов
- **Валидация:** Строгий временной сплит (80/20), без случайного сэмплирования (исключает data leakage)
- **Постобработка:** Множитель коррекции смещения, рассчитанный на валидационном окне

## 🚀 Как запустить
1. Загрузите `train_team_track.parquet` и `test_team_track.parquet` в среду выполнения
2. Запустите пайплайн в Google Colab
3. На выходе автоматически сформируются:
   - `submission_team_v3.csv` — файл для лидерборда
   - `README.md` — полная документация проекта
4. Для продакшен-развёртывания используйте стек: `FastAPI` + `Celery` + `Redis` + `MLflow`

## 📝 Бизнес-допущения
- Все транспортные единицы гомогенны (фиксированная вместимость `1.0` условной единицы отгрузки)
- Отгрузки равномерно распределены внутри 30-минутного интервала
- Задержки на погрузку/разгрузку учтены в горизонте 2ч
- Внешние факторы (погода, праздники, пробки) не учитываются в v1, но архитектура поддерживает их быстрое добавление

## 🔮 Пути развития и масштабирование
- 🌦️ **Внешние данные:** Интеграция погодных API, календаря праздников, данных GPS текущего положения транспорта
- 🧠 **Углубление моделей:** Переход на TFT / PatchTST / N-BEATS для длинных горизонтов и нестационарных рядов
- 🗺️ **Оптимизация маршрутов:** Интеграция с OR-Tools для минимизации пробега порожняком и cost-per-ton
- 📊 **MLOps:** MLflow для трекинга дрейфа данных, автоматический ретренинг по расписанию, A/B тестирование
- 🤝 **Бизнес-метрики:** Оценка через `Fill Rate`, `Deadhead Miles`, `Cost per Ton` vs базовое ручное планирование
